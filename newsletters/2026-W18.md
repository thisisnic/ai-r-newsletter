# AI Tooling & R Ecosystem Weekly Digest

A busy week for LLM tooling: Simon Willison's `llm` CLI got a significant overhaul, the R `tidyllm` package landed a substantial new release, and a few items worth watching surfaced on the security and policy fronts.

---

## LLM Tooling

**[LLM 0.32a0 is a major backwards-compatible refactor](https://simonwillison.net/2026/Apr/29/llm/#atom-everything)** — date not provided

Simon Willison's `llm` command-line tool reached a significant alpha milestone with a major internal refactor. The release is described as backwards-compatible, meaning existing plugins and workflows should continue to function. Worth keeping an eye on for anyone building on top of the `llm` ecosystem before it stabilises into a stable release.

**[Codex CLI 0.128.0 adds /goal](https://simonwillison.net/2026/Apr/30/codex-goals/#atom-everything)** — date not provided

OpenAI's Codex CLI gained a `/goal` command in version 0.128.0, allowing users to set a persistent objective that the agent works toward across interactions. This is a small but meaningful step toward more structured, session-aware agentic workflows in a terminal tool.

**[Quoting OpenAI Codex base_instructions](https://simonwillison.net/2026/Apr/28/openai-codex/#atom-everything)** — date not provided

Willison surfaces and comments on the base system instructions baked into OpenAI's Codex agent. Reading the defaults is a useful way to understand the constraints and assumptions the agent operates under — and to think about what you might want to override.

## R Packages

**[tidyllm 0.5.0](https://github.com/edubruell/tidyllm/blob/HEAD/NEWS.md#tidyllm-0-5-0)** — 2026-04-27

A substantial release for the R package that wraps multiple LLM providers in a tidyverse-friendly interface. The headline addition is a unified `.media` argument on `llm_message()` that accepts images, audio, video, and PDFs through a consistent set of constructors (`img()`, `audio_file()`, `video_file()`, `pdf_file()`). The OpenAI provider was rewritten to use the Responses API, unlocking reasoning effort controls, stateful conversations, built-in web search, and a code interpreter. A new `chat_completions()` provider covers any OpenAI-compatible endpoint without repurposing the `openai()` function. Default models across providers were updated to reflect recent releases.

## Security & Policy

**[Our evaluation of OpenAI's GPT-5.5 cyber capabilities](https://simonwillison.net/2026/Apr/30/gpt-55-cyber-capabilities/#atom-everything)** — date not provided

A third-party evaluation of GPT-5.5's capabilities in offensive cyber contexts. These assessments matter as a reference point for understanding what frontier models can and cannot do in adversarial settings, and for calibrating risk when deploying capable models in sensitive environments.

**[The Zig project's rationale for their firm anti-AI contribution policy](https://simonwillison.net/2026/Apr/30/zig-anti-ai/#atom-everything)** — date not provided

The Zig programming language project has articulated a clear policy against AI-generated contributions and explained their reasoning publicly. Whether or not you agree with the position, the writeup is a useful reference for open source projects thinking through how to handle AI-assisted code submissions.

**[Tracking the history of the now-deceased OpenAI Microsoft AGI clause](https://simonwillison.net/2026/Apr/27/now-deceased-agi-clause/#atom-everything)** — date not provided

A retrospective on the contractual clause between OpenAI and Microsoft that defined AGI and determined when Microsoft's access to OpenAI's technology would end. Now that the clause has been removed or superseded, this piece documents what it said and how it evolved — useful context for anyone following the commercial and governance structure of frontier AI labs.

## Infrastructure & Developer Tooling

**[What's new in pip 26.1 - lockfiles and dependency cooldowns](https://simonwillison.net/2026/Apr/28/pip-261/#atom-everything)** — date not provided

pip 26.1 adds native lockfile support and a "dependency cooldown" feature intended to reduce churn from transitive dependency updates. For teams managing Python environments alongside R workflows — common in data science shops running both stacks — this is a practical improvement worth knowing about.
