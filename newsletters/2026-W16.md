## LLM Tooling and Practice

**[A Field Guide to Rapidly Improving AI Products](https://hamel.dev/blog/posts/field-guide/)** — 2025-03-24
Hamel Husain's most practical post yet. The argument: most AI teams obsess over architecture and ignore measurement. The fix is error analysis, custom data viewers, and roadmaps counted in experiments rather than features. Includes real case studies showing date-handling failure rates dropping from 67% to 5% after systematic trace review. Worth assigning to any team that has a dashboard full of helpfulness scores but no idea what's actually broken.

**[LLM Evals: Everything You Need to Know](https://hamel.dev/blog/posts/evals-faq/)** — 2026-01-15
A dense FAQ distilled from teaching evals to 700+ engineers. Covers when to use binary vs. Likert scales (short answer: almost always binary), how to build trustworthy LLM-as-judge systems, synthetic data generation, annotation tool design, and the difference between guardrails and evaluators. Opinionated and useful.

**[Evals Skills for Coding Agents](https://hamel.dev/blog/posts/evals-skills/)** — 2026-03-02
Husain releases a set of slash-command skills for coding agents targeting eval pipelines — error analysis, judge validation, synthetic data generation, and annotation interface building. The framing is that coding agents now need domain knowledge about evals the same way they need docs. Pairs well with any MCP-enabled eval platform.

**[The Revenge of the Data Scientist](https://hamel.dev/blog/posts/revenge/)** — 2026-03-26
A talk-turned-post making the case that data science fundamentals — EDA, proper experimental design, calibrated classifiers, skepticism about labels — are exactly what AI teams are missing. The five eval pitfalls (generic metrics, unverified judges, bad experimental design, bad labels, automating too much) map directly to classic ML mistakes. Good for convincing a skeptical colleague that this work isn't new, just neglected.

**[Selecting The Right AI Evals Tool](https://hamel.dev/blog/posts/eval-tools/)** — 2025-10-01
Recorded panel review of LangSmith, Braintrust, and Arize Phoenix working through the same homework assignment. More useful than any feature comparison grid. Key criteria: friction from trace to playground, human-in-the-loop support, transparency vs. automation, and API quality for custom annotation apps.

## R Packages

**[ellmer 0.4.0](https://github.com/tidyverse/ellmer/blob/HEAD/NEWS.md#ellmer-0-4-0)** — 2026-04-13
Major release. `chat_claude()` is back as an alias for `chat_anthropic()`, following Anthropic's rebrand. `chat_openai()` now uses the responses endpoint. New `chat_openai_compatible()` for third-party APIs. Built-in web search tools for Claude, Gemini, and OpenAI. `parallel_chat()` and `batch_chat()` graduate from experimental. Several deprecated functions removed. If you're on an older version, review the lifecycle changes before upgrading.

**[tidyllm 0.4.0](https://github.com/edubruell/tidyllm/blob/HEAD/NEWS.md#tidyllm-0-4-0)** — 2026-03-17
Adds OpenRouter (300+ models) and llama.cpp providers, a `deep_research()` verb for long-horizon tasks via Perplexity, unified multi-turn tool loops across all providers, and `ellmer_tool()` for cross-ecosystem tool compatibility. The verb-based interface continues to mature. A solid option if you want a single consistent API across many providers.

**[chattr 0.3.0](https://github.com/mlverse/chattr/blob/HEAD/NEWS.md#chattr-0-3-0)** — 2025-08-18
Switches its LLM backend entirely to `ellmer`, dropping its own OpenAI and Databricks integrations. Streaming in the Shiny app is now handled via ellmer's stream functionality rather than a background process. Simpler and more maintainable as a result.

**[rollama 0.2.1](https://github.com/JBGruber/rollama/blob/HEAD/NEWS.md#rollama-0-2-1)** — 2026-03-23
Adds structured output support, custom headers for authentication, and custom output formats to the Ollama R client. Useful if you're running local models and want typed responses.

## Shiny and the R Ecosystem

**[Shiny for Python 1.6: Toolbars and OpenTelemetry](https://shiny.posit.co/blog/posts/shiny-python-1.6/)** — 2026-04-02
Two independent additions. Toolbars are compact input components designed for card headers, input labels, and chat interfaces — useful for adding controls without cluttering layouts. OpenTelemetry support lands as well, enabling traces of reactive execution, session lifecycle, and extended tasks with no code changes required. Set `SHINY_OTEL_COLLECT=reactivity` and point at Logfire or any OTLP backend.

**[OpenTelemetry + Shiny for R v1.12](https://shiny.posit.co/blog/posts/shiny-r-1.12/)** — 2025-12-10
The R-side OTel release. Automatically instruments session lifecycle, reactive updates, and extended tasks. The accompanying demo showing a GenAI weather app traced through Logfire — with model call time, tool execution, and streaming time all visible — is the clearest illustration of why this matters for LLM-powered apps.

**[Build Your First LLM App with Shiny for Python or R](https://shiny.posit.co/blog/posts/shiny-side-of-llms-part-3/)** — 2025-09-15
The third part of Posit's "Shiny Side of LLMs" series. Walks through building a Quarto presentation analysis app (DeckCheck) in both Python and R, covering async extended tasks, streaming, structured output, tool calling, custom loading states, interactive plots with ggiraph/Plotly, and deployment. More complete than most framework tutorials.

**[Tool Calling UI in shinychat](https://shiny.posit.co/blog/posts/shinychat-tool-ui/)** — 2025-11-20
shinychat now renders tool call requests and results as collapsible cards in the chat interface. Supports custom titles, icons, and display content (HTML tables, markdown). Chat bookmarking support also lands in this release, letting users restore conversation state from a URL.

**[querychat: Natural Language Queries for Shiny](https://shiny.posit.co/blog/posts/querychat-python-r/)** — 2026-01-22
querychat translates natural language questions into DuckDB SQL and returns reactive data frames. The implementation uses tool calling — the LLM proposes read-only SQL, querychat executes it — so results are reproducible and auditable. Works with in-memory data frames or external database connections. The SheScores demo shows replacing an entire sidebar of filters with a single chat input.

## Model Benchmarks (R Coding)

**[LLMs running on my laptop can drive coding agents now](https://simonpcouch.com/blog/2026-04-16-local-agents-2/)** — 2026-04-16
A follow-up to December's "local models are not there yet." Couch revisits local model performance for R coding agent tasks and finds meaningful improvement. No dramatic conclusion, but the trajectory is clear.

**[Claude 4 and R Coding](https://simonpcouch.com/blog/2025-05-27-claude-4/)** — 2025-05-27
Couch's benchmark of Claude 4 on his R eval dataset. Claude 3.7 Sonnet had been his daily driver; Claude 4 shows improvements worth noting for anyone doing R-heavy agent work.

## Commentary

**[Against "Brain Damage"](https://www.oneusefulthing.org/p/against-brain-damage)** — 2025-07-07
Ethan Mollick pushes back on the framing that AI use degrades thinking. The concern is real but the mechanism is more complicated than "AI makes you lazy." Worth reading before writing any internal AI usage policies.

**[Why I Stopped Using nbdev](https://hamel.dev/blog/posts/ai-stack/)** — 2026-01-18
Husain explains switching away from the notebook-centric literate programming tool he helped build. The core issue: AI coding tools were trained on conventional source code and get confused by nbdev's transpilation model. A rare honest post about abandoning something you built.
