## LLM Tooling

**[ellmer 0.4.0](https://github.com/tidyverse/ellmer/blob/HEAD/NEWS.md#ellmer-0-4-0)** — 2026-04-13

A significant release for the primary R interface to LLMs. `chat_claude()` is un-deprecated and aliased to `chat_anthropic()`, reflecting Anthropic's rebranding. `chat_openai()` now uses the responses endpoint and gains a new `chat_openai_compatible()` for third-party OpenAI-compatible APIs. New built-in web search and fetch tools for Claude, Gemini, and OpenAI; `batch_chat()` and `parallel_chat()` are out of experimental status. The `params()` helper gains `reasoning_effort` and `reasoning_tokens` for controlling thinking modes across providers.

**[tidyllm 0.4.0](https://github.com/edubruell/tidyllm/blob/HEAD/NEWS.md#tidyllm-0-4-0)** — 2026-03-17

Adds two new providers: OpenRouter (300+ models, fallback routing, cost metadata) and llama.cpp (local server support with chat, embeddings, reranking, and model management). Introduces `deep_research()` for long-horizon async research tasks via Perplexity's sonar model. Tool use is now unified across all supported providers, and ellmer tool definitions can be used directly via `ellmer_tool()`.

**[chattr 0.3.0](https://github.com/mlverse/chattr/blob/HEAD/NEWS.md#chattr-0-3-0)** — 2025-08-18

Switches entirely to ellmer for all LLM backend integration, dropping direct provider integrations (OpenAI, Databricks, LlamaGPT). The Shiny app now uses ellmer's streaming rather than a background process, which should reduce complexity and error surface.

## R Packages

**[Build Your First LLM App with Shiny for Python or R](https://shiny.posit.co/blog/posts/shiny-side-of-llms-part-3/)** — 2025-09-15

Part three of the Shiny Side of LLMs series walks through building "DeckCheck," a Quarto presentation analysis app. Covers async extended tasks, chaining LLM calls with structured output, tool registration, streaming, error handling with modals, and deployment. Side-by-side Python and R code throughout; practical for anyone integrating LLMs into production Shiny apps.

**[Shiny for Python 1.6 brings toolbars and OpenTelemetry](https://shiny.posit.co/blog/posts/shiny-python-1.6/)** — 2026-04-02

Two additions: a new `toolbar` component family for compact controls in card headers, input labels, and text areas; and built-in OpenTelemetry support. OTel traces session lifecycle, reactive updates, and all reactive expressions automatically with no code changes required. Pairs well with Logfire for GenAI latency debugging.

**[Tool Calling UI in shinychat](https://shiny.posit.co/blog/posts/shinychat-tool-ui/)** — 2025-11-20

shinychat now renders tool call requests and results inline in the chat interface as collapsible cards. Developers can customize titles, icons, and display content (HTML, markdown, or plain text) per tool. Also adds chat bookmarking support and improved programmatic control via `chat_mod_server()` return values.

**[Where Questions Become Queries: Meet querychat](https://shiny.posit.co/blog/posts/querychat-python-r/)** — 2026-01-22

Introduces querychat, which translates natural language to SQL (via DuckDB) and exposes the result as a reactive data frame in Shiny. Supports in-memory data frames, SQLite, and other DBI/SQLAlchemy-backed databases. The LLM never sees raw data or executes code directly—tool calling handles execution, keeping queries read-only and auditable. Drop-in replacement for filter sidebars in existing apps.

**[OpenTelemetry + Shiny for R v1.12](https://shiny.posit.co/blog/posts/shiny-r-1.12/)** — 2025-12-10

Shiny for R gains automatic OpenTelemetry instrumentation via the `{otel}` package. Traces cover session lifecycle, reactive update cascades, and individual reactive expressions. Controllable via `shiny.otel.collect` option; no changes to app code required to get started.

## LLM Evaluation and Practice

**[A Field Guide to Rapidly Improving AI Products](https://hamel.dev/blog/posts/field-guide/)** — 2025-03-24

Hamel Husain distills patterns from 30+ company engagements. Core argument: teams obsess over architecture and tooling while skipping measurement. Covers bottom-up error analysis (open coding → taxonomy → counts), building custom data viewers, empowering domain experts to write prompts directly, synthetic data generation along defined dimensions, and why AI roadmaps should track experiments rather than features. Dense with concrete examples.

**[Evals Skills for Coding Agents](https://hamel.dev/blog/posts/evals-skills/)** — 2026-03-02

A set of slash-command skills for coding agents (Claude Code, etc.) that guard against common eval mistakes. Skills include error analysis, synthetic data generation, LLM-as-judge prompt writing, evaluator calibration, RAG evaluation, and building annotation interfaces. Designed to complement eval platform MCP servers, which handle access but not methodology.

**[LLM Evals: Everything You Need to Know](https://hamel.dev/blog/posts/evals-faq/)** — 2026-01-15

An FAQ compiled from teaching 700+ engineers. Covers why binary pass/fail beats Likert scales, why generic off-the-shelf metrics are actively harmful, how to sample traces efficiently, when to outsource annotation (rarely), and how to evaluate agentic and multi-turn workflows. Opinionated and direct; useful as a reference for teams setting up eval pipelines.

**[The Revenge of the Data Scientist](https://hamel.dev/blog/posts/revenge/)** — 2026-03-26

Conference talk transcript arguing that classical data science skills—EDA, experimental design, classifier validation, label quality—are exactly what's missing from most LLM product development. Five failure patterns: generic metrics, unverified judges, bad experimental design, bad labels, and over-automation. The harness around an LLM system is where data science lives.
